ARP — 3 propriétés clés Très fréquent
Requête ARP (broadcast)

MAC destination = FF:FF:FF:FF:FF:FF
→ Envoyée à tous les équipements du LAN
→ Le switch inonde tous ses ports

Réponse ARP (unicast)

Renvoyée uniquement à la source
→ L'adresse MAC source dans l'en-tête Ethernet permet l'identification

ARP statique

Entrée IP↔MAC saisie manuellement → utile pour les serveurs fréquemment utilisés (évite les requêtes répétées)

IPv6 : plus d'ARP !

Remplacé par ICMPv6 Neighbor Discovery
→ Neighbor Solicitation (requête)
→ Neighbor Advertisement (réponse)

⚠ Piège classique
"Pourquoi le switch inonde lors d'une requête ARP ?" → Parce que la destination MAC est FF:FF:FF:FF:FF:FF (broadcast), pas parce que la source est un broadcast.
Comportement du switch
SituationComportement
MAC destination connue dans la tableEnvoie uniquement vers le port concerné
MAC destination inconnueInonde tous les ports (sauf port d'entrée)
Adresse de diffusion (broadcast)Inonde tous les ports (sauf port d'entrée)

Compression IPv6 — règles
Règle 1 — Zéros initiaux

Supprimer les zéros initiaux dans chaque groupe
0090 → 90 · 0000 → 0

Règle 2 — Double deux-points

:: remplace une seule série continue de groupes à zéros — ne peut être utilisé qu'une fois !

⚠ Piège
3FFE:1044:0000:0000:00AB:0000:0000:0057
❌ 3FFE:1044::00AB::0057 — deux :: interdits !
✅ 3FFE:1044:0:0:AB::57 — :: sur la 2e série de zéros seulement
EUI-64 & SLAAC
EUI-64

Crée l'ID d'interface (64 bits) à partir de l'adresse MAC (48 bits)
→ Insère FF:FE au milieu de l'adresse MAC
→ Inverse le 7e bit

SLAAC

Configuration automatique sans serveur DHCP
→ Le routeur envoie le préfixe réseau
→ L'hôte génère son propre ID d'interface (via EUI-64 ou aléatoire)

TCP vs UDP — comparaison directe Très fréquent
CritèreTCPUDP
ConnexionOrienté connexion (handshake 3 étapes)Sans connexion

✓ UDP quand ?
UDP est préféré quand : la vitesse prime sur la fiabilité, ou quand l'application gère elle-même la fiabilité, ou quand une petite perte de données est acceptable (streaming, VoIP)
Fenêtre TCP & numéros de séquence Calculs aux QCM
ℹ Logique
La taille de la fenêtre = quantité de données que la destination peut traiter (pas la source). L'accusé de réception (ACK) contient le numéro du prochain octet attendu.

Exemple : fenêtre 6000 octets, paquets de 1500 octets

Après réception deACK envoyé par le serveurLogique
1 paquet (octets 1–1500)ACK = 15011500 + 1 = 1501
2 paquets (1–3000)ACK = 30013000 + 1 = 3001
3 paquets (1–4500)ACK = 45014500 + 1 = 4501
⚠ Piège
Après 2 paquets de 1500 octets → ACK = 3001 (pas 3000, pas 4500). L'ACK indique le prochain octet attendu = dernier reçu + 1.

P2P vs Client-Serveur
Client-Serveur

Serveur dédié + clients
Ex : requête DNS, navigation web, FTP
Un équipement joue un seul rôle à la fois

P2P (Peer-to-Peer)

Pas de serveur dédié — ressources décentralisées
Ex : BitTorrent, eMule, eDonkey, Bitcoin
Un équipement peut être client ET serveur simultanément
